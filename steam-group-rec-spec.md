# Steam 群体多人游戏推荐 —— 开发规格文档

> 面向 Codex / Claude Code 的开发规格书。本文档记录已确定的产品决策、算法逻辑、数据流与边界情况，可据此直接开工。
> 文中标注 **[待勘查]** 的项，是依赖 Steam 真实 API 返回、必须在编码前手动跑通验证的点，不要基于假设编码。

---

## 0. 一句话定义

给一小群（几人到十几人）Steam 玩家，基于他们**各自的库存与游玩时长**，建模出这桌人的**共同口味**，从中推荐**他们都没有、值得一起联机玩**的游戏，并给出「为什么适合你们这一桌」的可信理由。

---

## 1. 产品定位与核心决策

### 1.1 核心功能 vs 加分项

| 模式 | 定位 | 说明 |
|---|---|---|
| **B：发现新游戏（主功能）** | 核心 | 从「这桌人都没有」的联机游戏里，按共同口味推荐值得一起买来玩的。**整个项目的核心价值。** |
| A：共有库存选今晚玩什么 | **不做** | Steam 自带类似能力；且「有但没玩」多半是玩不进去，价值不大。明确排除。 |

### 1.2 「新」的定义 —— 两档，UI 上分开

1. **主线档（默认）**：他们**错过的好游戏**——已发售、有完整口碑数据（Steam 近期好评率、评论数）。稳、可信、有「你们居然都错过了它」的故事性。
2. **尝鲜档（可选勾选）**：未发售 / 抢先体验 / Demo 的全新内容。明确告知用户「这些尚未经过时间检验」。
   - **⚠️ 尝鲜档走独立打分逻辑，不是主逻辑打折**（详见 §4.4）。

> 两档置信度不同，UI 必须分区展示、不可混排。

### 1.3 我们相对短视频推新博主的护城河

不是「知道有这个新游戏」（信息公开、抢不过博主），而是 **「群体口味 × 新游戏」的精准匹配 + 可解释理由**。
博主是广播（对所有人喊「好玩」）；我们知道**这一桌具体是谁**，能说出：

> 「这游戏适合你们这桌：因为老张爱的 X、小李爱的 Y 它同时占了；你们没人有它；联机刚好支持 4 人。」

**这句「为什么」就是信任来源，是产品命根子。** 所有设计都要服务于「能讲出这句话」。

## 2. 系统架构与信任模型

### 2.1 密钥与数据的存放原则

- 每个用户**自带** AI key 和 Steam Web API key，避免共享额度被封。
- key 存**客户端本地**，服务器**不落库、不写日志、不留存**。
- 库存时长等用户数据**不入库**，仅在客户端 web 短暂缓存（1 天 ~ 数天）。

### 2.2 两条数据线分开处理（重要）

| 数据线 | 是否过服务器 | 原因 |
|---|---|---|
| **AI API 调用** | **不过服务器**，浏览器直连 | Anthropic API 支持带 header 从浏览器直连；AI key 完全不碰服务器，信任面最小。 |
| **Steam Web API 调用** | **必须过服务器代理** | Steam 接口不发 CORS 头，浏览器直连会失败。服务器退化为**无状态转发代理**。 |

> 服务器职责：仅转发 Steam 请求 + 维护共享缓存（见 §5）。不存 key、不存用户数据。

### 2.3 安全基线（受众仅几十人、低频，不过度设计）

- 首要目标：**服务器被攻破也不能泄露 key**——做法就是服务器里根本没有 key。
- localStorage 对 XSS 透明，上严格 **CSP**；敏感 key 优先放内存 / sessionStorage。
- 出大问题直接关机 / 重装（服务器无状态，无数据损失顾虑）。

## 3. 数据来源与候选池

### 3.1 候选池三个来源（可解释分层）

推荐结果里每个游戏都要标清楚它**因何进入候选**，作为对抗「AI 瞎推」质疑的透明度武器：

| 来源 | 说明 | 标记 |
|---|---|---|
| **口味匹配** | 与这桌人共同口味向量高度贴合 | 「因为对味」 |
| **获奖 / 提名** | 命中 TGA 等年度获奖提名名单 | 「TGA 2024 提名」等 |
| **近期热门 / 新品** | 最近发售或口碑上升的多人新游 | 「最近火」 |

### 3.2 候选枚举与过滤的铁律

> **枚举和过滤用结构化数据（不会错）；判断和排序用 AI（会错但影响可控）。绝不让 AI 决定「候选存不存在」。**

**候选枚举**（结构化来源，禁止靠 AI 生成候选，会幻觉 appid / 报错年份）：

- **[待勘查]** 最近 N 个月的多人新品 / 热销榜。**Steam 没有干净的「近 3 个月多人新游戏」官方接口**，此为 B 的进料口，必须在开发早期确定走法（store search 页带参、SteamDB、或其他）。**这是最悬空、最高优先级的勘查项。**
- 获奖 / 提名名单（近几年多人类，静态小表，见 §3.4）。

**硬过滤**（AI 不参与）：

1. **联机过滤**：用 `appdetails` 的 `categories` 里的 Co-op / Online Co-op / Multi-player 类目，非多人直接踢。**[待勘查]** 具体 category id。
2. **拥有过滤**：见 §3.3。
3. **质量门槛**：Steam 近期好评率过低 / 评论数太少的先不推。

### 3.3 拥有过滤规则（不是按热度排除，是按「这桌人是否已拥有」）

- **主规则**：在**这桌人库存并集**里出现过的游戏，全部排除。
- **不要**用「知名度 / 热度」一刀切——有些大众「该玩都玩了」的游戏，对**这桌特定的人**恰是盲区，且「大家居然都没买过的好游戏」转化率最高（集体补票、一人补票全队开黑）。
- 兜底：可保留「太国民级（如评论数百万级）默认不推」的软规则，但**主过滤器是库存并集，不是热度**。
- **边界情况**：五人里已有三人拥有 → 它未被全员拥有，仍进候选，但推荐时**打标**「你们中 3 人已拥有，补票即可开黑」，作为转化钩子，**不要当普通新游戏推**。

### 3.4 信任锚：获奖 / 提名名单

- TGA 及各类年度获奖 / 提名是**有限、静态、权威**的，一次性存成小表（近几年）。
- 双重用途：① 作为**候选池的一等高质量来源**（近年多人类提名 = 一份精选候选清单）；② 命中时**打标背书**（比任何 AI 生成理由都可信，因为是第三方权威）。

### 3.5 评分来源的取舍

- **以 Steam 玩家近期好评率为主**：直接反映「现在」的实际联机体验和社区状态。
- 媒体分（IGN / Metacritic）**仅辅助**：媒体分与玩家分在联机游戏上经常打架（媒体高分但社区因氪金 / 服务器 / 外挂骂声一片）。**别让媒体分盖过 Steam 近期评价**，否则会推出「评分高但现在没人玩 / 服务器已死」的游戏。

### 3.6 免费游戏（易漏的坑）

- 拉库存时**必须传 `played_free_games`**（[待勘查] 确认参数名），否则丢掉 Dota2 / CS / Warframe 等时长——而这些恰是**最强的多人信号**，漏了等于砍掉多人推荐最重要的数据源。

### 3.7 社区搜索（软信号，可有可无、量小）

- 仅对最终 **top 几名**做「近期塌房 / 社区氛围」这类**无法结构化**的软判断复核。
- 是排序层最后一层调味，**不是数据管道主干**。频率极低，符合「不想频繁调搜索」的诉求。

## 4. 口味模型（B 的引擎，核心算法）

### 4.1 标签来源

- **[待勘查] 最高优先勘查项之一**：用 Steam **store tags（用户投票标签）**，取每个游戏**票数前 15–20 个**（Steam 本身也这么截断），兼顾细粒度与降噪（滤掉玩梗投票）。
- **⚠️ 地基风险**：整个口味模型依赖「按票数取前 N」。若 Steam 公开接口**只给标签名、不给票数**，则「截断前 20」无法做，需改为全量标签或换降噪方式，**并连带影响打分**。这是**地基级返工风险**，必须在编码前先瞟一眼 store tags 的票数字段是否可取。

### 4.2 打分公式（已修正三个失真坑）

对**每个人、每个标签**：

```
个人标签深度信号 = Σ sqrt(该标签下每个游戏的时长)
```

- `sqrt`：压缩长尾，避免单个游戏（如 PUBG 3000h）绑架整个标签的品类分。
- `Σ`（对该人名下该标签的多个游戏求和）：体现广度——多个游戏都投入才算真爱品类。

然后**在这个人内部归一化**：

```
个人标签权重 = 个人标签深度信号 / 该人所有标签深度信号之和
```

- 归一化目的：消除重度玩家 vs 休闲玩家的绝对时长差（可达 8 倍），让「我这点时间里 70% 花在 co-op」能和硬核玩家平等对话。

最后 **group 标签分**：

```
group 标签分 = 所有人「个人标签权重」的平均值   # 平均，不是加总
```

- **必须平均、禁止加总**：加总会让重度玩家的口味独裁，其他人形同虚设。group 场景的灵魂是照顾每个人。

> 三个坑的由来（防止实现时改回错误版本）：
> 1. **禁止 `总时长 × 游戏数 × 系数`**：数量被算两遍会指数爆炸，让「喜加一仓鼠」碾压「真爱深度玩家」。
> 2. **时长必须 sqrt/log 压缩**：否则单个极端时长游戏绑架标签分。
> 3. **必须先个人内归一化再跨人平均**：否则重度玩家独裁。

### 4.3 聚焦 vs 发散：用分布形状决定策略，不要预设阈值

算完所有 group 标签分后看分布：

- **断崖式收敛**（第一名远高于第二名，占总分一大半）→ 口味高度一致，**聚焦该品类**推。
- **前几名接近**（口味发散 / 多元）→ 做**多标签组合命中**：一个游戏同时占据前几高分标签中的 ≥2 个，比只占第一名的更该推。

> 用「分布形状」判断，比拍脑袋定固定阈值稳。

### 4.4 标签组合优先于单标签

- Steam 游戏从不是单标签，口味也不是。真正信号常是**标签共现组合**（如「FPS × Roguelike 交叉带」），而非孤立单标签。
- 打分时除单标签分外，**给「命中多个高分标签的游戏」额外加权**。

### 4.5 用户勾选 / pass 交互（当成口味向量调整，不是黑白名单）

- **勾选某标签** = 对该标签**大幅加权**。
- **pass 某标签** = **大幅降权甚至归零其单标签贡献**。
- **⚠️ 禁止做成硬过滤 / 一票否决**：因为游戏是多标签的，硬过滤会误伤好游戏。
- **组合豁免规则**：`pass = 仅当游戏只有该标签时排除；当它与其他 in（勾选）标签共现时豁免`。
  - 例：用户 pass 了「生存建造」但勾了「FPS」→「FPS + 生存建造」的游戏仍可推。

### 4.6 尝鲜档（纯新游戏）独立打分逻辑

- **不是主流程打折**。纯新 / 未发售 / Demo **没有 Steam 玩家好评率、标签可能不全** → 主打分的支柱（近期好评率）为 null。
- 走**独立轻逻辑**：只靠**标签匹配 + 获奖提名 + 媒体前瞻**，明确无玩家评价支柱。
- **实现提醒**：若硬塞进主打分管道，会得到一堆 null 字段。必须单独一条路径。

## 5. 缓存策略

### 5.1 缓存围绕「游戏属性」组织，不是围绕「榜单」（关键）

- **错误做法**：缓存一个全局「昨天有哪些新游戏」的榜单——这样讲不出「老张爱 X、小李爱 Y 这游戏正好都占」，因为榜单里没有 per-game 标签维度。
- **正确做法**：缓存**每个候选游戏的属性包**（per-game record）：
  - 标签、category（几人联机）、好评率、评论数、发售 / 更新时间。
- **缓存粒度决定能不能讲出「为什么适合你们」那句话** —— 这是能力边界问题，不只是性能优化。

### 5.2 TTL 分层

| 数据 | 变化性 | 缓存策略 |
|---|---|---|
| 标签、category、是否多人、类型 | 几乎永不变 | 长期缓存 |
| 好评率、评论数、更新时间 | 缓慢变 | 几天 TTL |

### 5.3 维护方式：增量维护「属性库」，不是维护「榜单」

- 维护一个「最近 N 个月的多人游戏属性库」（约几千个游戏量级）。
- 定时任务（如每天一次）**增量**：补昨天新增的多人新游、刷新一批临期的好评率。
- 冷门 / 老牌大热之外的游戏**现用现拉**，拉到即补进库。
- **跨用户共享**这份缓存：一台服务器 IP，一两周被几十人（每人几百个大量重叠 appid）打一次，且大部分命中缓存 → **摸不到限速的边**。

> IP 限速的真解是**缓存**（候选集有界、大量重叠、多为静态数据），不是「拉长间隔 / 每天更新一点」这种治标节奏调整。

## 6. AI 的职责边界

### 6.1 分工：算法定候选与粗排，AI 只做精排 + 写理由

- 候选可能几百上千，**不可能全塞进 context**。
- **确定性算法先用口味分砍到 top 20–30**，AI 只对这一小批做精排和写推荐语。
- 好处：AI 幻觉空间被锁死在小池子里，**不指望 AI 从上千个里挑**。

### 6.2 理由必须基于真实数据，禁止自由发挥

- AI 那句「为什么适合你们」**只能引用喂给它的真实事实**（老张的标签 / 时长、老李的偏好、口碑数据、获奖标记）。
- **禁止让 AI 自由编造理由**——一旦用户发现推荐理由是瞎编的，信任瞬间归零。
- Prompt 需硬约束：仅允许引用系统提供的标签 / 时长 / 口碑 / 获奖事实。

### 6.3 AI 输入 / 输出契约

- **输入**：口味向量（group 标签分 + 分布形状信息）、候选 top 列表（每个含标签、category、近期好评率、评论数、获奖标记、哪些人已拥有）、用户勾选 / pass。
- **输出**：带**可解释理由**的排序结果，每个游戏标注**候选来源分层**（对味 / 获奖 / 最近火）与**特殊标记**（如「3 人已拥有，补票开黑」）。

## 7. 边界情况与降级（Degenerate Cases）

### 7.1 单个 ID 数据不足 / 私密 —— 踢出该人，不停全队

- 判定单位是**「团队还剩几个有效人」**，不是「有没有人不合格」。
- 单个人私密 / 库存太薄（如仅 3 个游戏、总时长几小时）→ **踢出本次计算并明确告知**（「老李数据不足 / 档案私密，已排除，其余 N 人继续」）。
- 仅当有效人数少到无法谈「共同口味」（如只剩 1 人）→ 整体停止，明确提示「数据不足，无法推荐」，让用户删掉该 ID 或改库存状态 / 隐私设置。
- 隐私要求写在 web 上，强制满足，不满足不可用。

### 7.2 这桌人口味无交集 —— 必须优雅处理（出现频率比想象高）

- 症状：归一化平均后所有标签分都很平（一坨均匀的泥），选不出主品类。算法不报错，但会输出一堆平庸大众多人游戏，推荐质量归零。
- 需明确降级策略（三选一或组合，实现时定）：
  1. 坦诚告知「你们口味差异较大，以下是几个最大公约数」；
  2. 退化成分别照顾（给每人各推一个）；
  3. 按子团体拆分推荐。

### 7.3 拥有过滤的边界

- 全员拥有 → 排除。
- 部分人拥有（如 5 人中 3 人）→ 进候选但打「补票开黑」标（见 §3.3）。

## 8. 最小可行闭环（MVP 数据流）

```
1. 输入多个 Steam ID（支持 vanity URL → SteamID64 解析）
2. 拉每人库存 + 时长（含 played_free_games）；私密/太薄的踢出并告知
3. 有效人数校验（不足则停，见 §7.1）
4. 从库存提取每人标签深度信号 → 个人内归一化 → 跨人平均 → group 标签分 + 分布形状
5. 候选枚举：近 N 月多人新品 + 获奖提名名单（结构化来源）
6. 硬过滤：联机过滤(category) + 拥有过滤(库存并集) + 质量门槛(好评率/评论数)
7. 用 group 标签分（含组合加权、勾选/pass 调整）对候选粗排 → top 20–30
8. AI 精排 top 20–30 + 生成基于真实数据的可解释理由
9.（可选）对最终 top 几名做社区氛围/塌房搜索复核
10. 分区展示：主线档 + 尝鲜档；每项标注候选来源分层与特殊标记
```

## 9. 开工前必须先做的事（不要跳过）

> 以下依赖 Steam 真实返回，**在写业务逻辑前手动跑通、打出真实 JSON 看字段**，否则会基于想象编码然后返工。

**P0（地基级，错了要大改）：**
1. **store tags 能否拿到票数** —— 决定「截断前 20」能否成立，影响整个 §4 打分模型。
2. **「近 N 月多人新游戏」候选如何枚举** —— Steam 无干净官方接口，是 B 的进料口，断了整个产品空转。

**P1（局部，错了局部返工）：**
3. `appdetails` 里多人 category 的具体 id（§3.2 联机过滤）。
4. 库存接口 `played_free_games` 的确切参数名（§3.6）。
5. `appreviews` 近期好评率 / 评论数字段与 `filter=recent` 用法（§3.5）。
6. 更新时间来源（`GetNewsForApp` 或商店更新历史）。

> 其余字段车到山前再调；但 P0 两项务必先验证，它们会反向改动已定的模型与候选逻辑。

## 附录：一句话原则清单（防止实现时走偏）

- 口味是产品引擎，引擎偏一度、输出偏十里。
- 枚举/过滤用结构化数据，判断/排序用 AI，绝不让 AI 决定候选存不存在。
- 打分：sqrt 压时长 → 个人内归一化 → 跨人平均。禁止乘法堆叠、禁止加总。
- 用分布形状决定聚焦还是组合，不预设阈值。
- pass 是单标签否决，组合可豁免；勾选/pass 是向量调整不是黑白名单。
- 缓存 per-game 属性包，不缓存榜单。
- 拥有过滤按库存并集，不按热度。
- AI 理由只准引用真实数据，禁止自由发挥。
- 尝鲜档独立逻辑，不是主流程打折。
- 降级看「有效人数」，不看「有没有人不合格」。
